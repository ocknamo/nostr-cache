# コンセプト / 目的（プロジェクトのビジョン）

> このドキュメントは Nostr Cache プロジェクトが「何を実現しようとしているのか」を
> 定義する。各パッケージの設計判断や未実装項目（[doc/TODO.md](./TODO.md)）は、
> すべてここに書かれた目的に照らして評価される。

## 1. 出発点となるアイデア

Nostr クライアントのキャッシュを**完全に**行うには、キャッシュ専用のデータ構造を
クライアントに作り込むのではなく、**クライアント層で動く Nostr リレー実装そのもの**を
用意し、それをキャッシュとして動かすのが最も筋が良い、というのが本プロジェクトの
出発点である。

クライアントから見たキャッシュが「ただのリレー」であれば、

- クライアント側に Nostr 専用のキャッシュ層を新規に実装する必要がない
- クライアントは普段どおり WebSocket でリレーと話すだけでよい
- リレーの語彙（NIP-01 の `EVENT` / `REQ` / `CLOSE` ...）がそのままキャッシュの
  インターフェースになる

というメリットが得られる。

## 2. 同一コア・多環境という設計

中核となるリレー実装（`packages/cache-relay` の `NostrCacheRelay`）は、
**ストレージ**と**トランスポート**をアダプタとして差し替えられる。これにより
同じリレーコアを次の 2 つの形態で動かす。

| 形態 | トランスポート | ストレージ | 役割 |
|---|---|---|---|
| サーバ | `WebSocketServer`（Node.js / `ws`） | IndexedDB 互換（永続化が目標） | **普通の Nostr リレー**として動く |
| Web クライアント | `WebSocketServerEmulator`（`globalThis.WebSocket` をインターセプト） | IndexedDB（ブラウザ） | **ローカルキャッシュリレー**として動く |

- **サーバで実行すれば普通のリレーになる。** 通常の Nostr リレーとして外部公開できる。
- **Web クライアントでは、ページ内の WebSocket をインターセプト**（あるいは同等の
  手段）して、クライアントの通信をローカルのリレー実装に向ける。クライアントから
  見ると「リレーに接続している」だけだが、実体はローカルのキャッシュである。

## 3. 最終的に目指す「完全なキャッシュ」の姿

ローカルリレーが自身に保存済みのイベントを返すだけでは、クライアントの通信を
肩代わりするキャッシュにはならない。最終形では、ローカルリレーが**上流の実リレー
群の手前に透過的に挟まる**ことを目指す。

```
+------------------+      WebSocket       +------------------------+      WebSocket      +-------------+
| Nostr クライアント | -- (インターセプト) --> | ローカルキャッシュリレー   | -- (キャッシュミス時) --> | 上流実リレー群 |
| (普段どおり接続)    | <------------------- | (cache-relay コア + IndexedDB) | <------------------- | nos.lol 等   |
+------------------+                      +------------------------+                     +-------------+
```

- **リードスルー**: `REQ` を受け取り、ローカルに十分な結果がなければ上流リレーへ
  問い合わせ、得たイベントをローカルへ充填してからクライアントへ返す。
- **ライトスルー**: `EVENT`（投稿）をローカルへ保存しつつ、上流リレーへも転送する。
- **透過性**: クライアントは自分がキャッシュ越しに通信していることを意識しない。

これらが揃って初めて「クライアントキャッシュを完全に行う」という当初の目的が
達成される。

## 4. 現状とのギャップ（2026-06 時点）

本リポジトリは上記ビジョンの**途中段階**にある。実装済み・未実装の詳細は
[doc/TODO.md](./TODO.md) を参照。要点のみ:

- ✅ リレーコア（NIP-01 のメッセージ処理、replaceable イベント対応、ストレージ、
  トランスポート、購読管理）は実装済み（NIP-02 は kind 3 を replaceable として扱う範囲）。
- ✅ サーバ形態・ブラウザ形態それぞれのトランスポートが存在する。
- 🔶 サーバは現状 `fake-indexeddb`（インメモリ・非永続）で、永続リレーには未到達。
- ❌ Web クライアント（`packages/web-client`）はまだ `WebSocketServerEmulator` を
  経由しておらず、生の WebSocket で直接リレーに接続している。インターセプトの
  エンドツーエンドが未配線。
- ❌ 上流リレーへのリードスルー / ライトスルー（第3節の透過キャッシュ）は未実装。
  現状のローカルリレーは「自分が持つイベントのみ返す独立リレー」にとどまる。
